# RLT Architecture - Correct Implementation

## Overview

The RLT (Reinforcement Learning Teachers) methodology trains a **student model** to solve problems by learning from high-quality explanations generated by a **teacher model**.

## Key Components

### 1. Teacher Model (Claude Sonnet 4)
- **Role**: Generates step-by-step explanations
- **Implementation**: Claude API (`src/teachers/claude_teacher.py`)
- **Trainable**: NO - uses pre-trained Claude via API
- **Function**: Given a question and answer, produces detailed reasoning

### 2. Student Model (HuggingFace)
- **Role**: Learns to solve problems using teacher explanations
- **Implementation**: Local model (`src/models/optimized_model.py`)
- **Trainable**: YES - this is what we train!
- **Models**: Llama-3.2-3B, Mistral-7B, etc.
- **Function**: Takes question + explanation → produces answer

### 3. Evaluator
- **Role**: Assesses student understanding for reward computation
- **Implementation Options**:
  - Local model: `src/rewards/student_evaluator.py` (lower cost)
  - Claude API: `src/rewards/claude_evaluator.py` (highest quality)
  - Hybrid: Mix of both (balanced cost/quality)
- **Function**: Measures how well student can solve problems (rSS score)

### 4. Reward Function
- **Components**: 
  - rSS: Solution score (how well student solves problems)
  - rKL: KL divergence (diversity of explanations)
- **Formula**: `reward = rSS - λ * rKL`

## Training Flow

```
1. Dataset provides (question, answer) pairs
   ↓
2. Teacher (Claude) generates explanations
   Example: "What is 15 × 8?"
   Teacher: "To find 15 × 8, I'll use the distributive property..."
   ↓
3. Student receives (question, explanation, answer) tuples
   Student learns: Given question + explanation → produce answer
   ↓
4. Evaluator measures student performance
   How well can student solve similar problems?
   ↓
5. Reward computed based on student understanding
   High reward = Student learned well from explanation
   ↓
6. GRPO updates STUDENT model parameters
   Student gets better at using explanations to solve problems
```

## Common Misconceptions

### ❌ WRONG: "Student evaluates teacher explanations"
The student doesn't evaluate - it LEARNS from explanations. The evaluator component measures student performance.

### ❌ WRONG: "Training the teacher model"
We never train the teacher (Claude). We only train the student model.

### ❌ WRONG: "Student model just mimics teacher"
The student learns to solve problems USING teacher explanations, not just copy them.

## Implementation Details

### Correct Training Loop (Simplified)
```python
# 1. Teacher generates explanation
explanation = teacher.generate_explanation(question, answer)

# 2. Student learns from explanation
student_input = f"Question: {question}\nExplanation: {explanation}\nAnswer:"
student_loss = student_model.compute_loss(student_input, target=answer)

# 3. Evaluator measures understanding
student_score = evaluator.evaluate(student_model, test_questions)

# 4. Compute reward
reward = student_score - lambda * kl_divergence

# 5. Update STUDENT model
optimizer.step()  # Updates student_model parameters only!
```

## Why This Architecture?

1. **Leverages Strong Teacher**: Claude provides high-quality explanations
2. **Efficient Student Training**: Smaller models learn from better explanations
3. **Dense Rewards**: Every explanation provides training signal
4. **Scalable**: Can train many different student models

## Results

With this architecture:
- 7B student models can outperform 70B+ models
- Training is more sample-efficient
- Students learn reasoning, not just memorization

## Key Insight

The magic of RLT is that a small student model can learn to solve problems effectively by studying high-quality explanations from a strong teacher, rather than trying to learn everything from scratch.
